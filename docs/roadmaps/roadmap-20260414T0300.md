## 已知限制

| 限制 | 影响 | 可能的解法 |
|------|------|-----------|
| obs/session 比率受空 session 稀释 | 每次 Claude Code 启动都创建 session，即使没有实质工作，拉低比率 | 在审计中过滤掉 <1 prompt 的 session，或标记"空 session"不计入 |
| SDK Agent 提取是 AI 调用 | 每个 observation 需要一次 AI API call，受 rate limit 和延迟影响 | 批量处理（3-5 tool calls per AI call）或本地模型替代 |
| PostToolUse hook 是外部进程 | 每次 tool call spawn 一个 node 进程，开销大且可能被 Claude Code 限速 | 长连接（WebSocket/IPC）替代 spawn-per-call |

## 建议的下一个迭代

1. **修复 PostToolUse hook 触发率** — 这是 E-201 的根本限制。hook 不触发，任何下游优化都无效。需要调查 Claude Code 是否有 hook 调用限速，或改用 IPC/长连接。
2. **GateKeeper 状态持久化** — 小改动大收益。持久化 lastCompilationTime 让 24h cooldown 真正生效，避免无意义的重复编译。
3. **Transcript recovery 集成** — 949 个 JSONL 文件未处理。接入现有的 ClaudeCodeTranscriptProcessor 可以大幅增加历史 observation 数据。
